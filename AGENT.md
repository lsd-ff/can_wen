# CanW 项目状态与接手说明

更新日期：2026-07-14

本文档以当前代码、数据库迁移和本轮验证结果为准。它优先说明用户端已具备的能力、明确未纳入本迭代的能力，以及正式上线前才需要补齐的运行条件。

## 当前结论

CanW 已完成面向家蚕养殖者的问诊、养殖过程管理和多用户社区闭环。排除已明确延期的真实 RAG、知识图谱、可追溯诊断证据和长期记忆后，用户端不再需要为“补齐基础功能”继续开发，可用于简历展示、毕业设计演示和本地试用。

后续用户端只处理真实使用反馈、缺陷修复与体验微调；正式公开上线前再接入外部基础设施和多实例运行保障。本项目当前不是“已经商业化上线”的声明，任何文档和演示都不得将延期能力描述为已实现。

## 用户端能力总览

### 1. 账户、身份与个人数据

- 手机号和邮箱验证码登录、刷新令牌、会话管理与退出登录。
- 个人资料、头像和显示信息维护。
- 用户模型配置、连通性测试与默认模型选择；用户 API Key 以 AES-GCM 密文保存，并兼容历史 `v1` 密文。
- 账户数据导出：身份资料、登录活动、项目、对话、消息、多模态结果、附件元数据、养殖记录和社区数据。
- 账户删除会同步处理所属数据；对象存储未配置时会安全拒绝删除，避免只删数据库而遗留附件。

### 2. AI 问诊

- 新建、续聊、编辑、删除、重新生成、反馈、归档、置顶、项目归属和公开分享。
- 服务端搜索覆盖会话标题、自动摘要与消息正文；用户首条有效问题会自动生成更易识别的中文对话标题。
- AI 回复以富 Markdown 呈现，支持标题、粗体、列表、引用、行内代码、代码块与一键复制；用户消息保持普通文本，避免内容被错误解析。
- 输入框支持 `Enter` 发送、`Shift + Enter` 换行，并避免中文输入法组词时误发送。
- 图片、视频、文档和语音材料上传；PDF 正文会提取并保存为多模态上下文，其他文件保留受控对象 URL 供模型读取。
- 问诊中可直接录入结构化养殖信息：日期、蚕龄、温湿度、给桑、用叶量、异常/死亡数量和现场备注，并随本次问诊发送。
- 会话可导出为 Markdown；问诊可保存为养殖病例，或生成待发布的社区草稿。

### 3. 项目与全局检索

- 支持将问诊归入项目，按项目组织持续的养殖问题与资料。
- 应用内搜索可定位相关问诊、用户和社区内容，并跳转到对应业务页面。
- 侧栏对话标题以摘要展示，减少大量“新问诊”或首句标题造成的识别困难。

### 4. 养殖工作台

- 养殖场管理：创建、编辑、删除与当前养殖场切换。
- 批次管理：新建、编辑、删除、切换批次；显示品种、起养数量、预计上蔟时间和养殖阶段。
- 每日记录：记录环境、饲养、健康与异常数据；默认展示最近五条，展开后固定高度滚动查看全部，支持收起和按日期查看。
- 病例台账：新增、编辑、筛选、搜索、跟进和关闭病例；支持从问诊导入或由养殖病例生成社区草稿。
- 养殖进度、健康提醒和实用计算在当前批次范围内呈现，避免不同批次数据混杂。
- 社区中的病例内容可保存到用户私有养殖台账，形成“社区经验 → 自己的养殖记录”的闭环。

### 5. 家蚕交流社区

#### 内容创作与消费

- 发布帖子、保存草稿、编辑和删除自己的帖子；支持图片、视频、文档附件、封面图和素材排序。
- 信息流支持推荐、关注、话题关注、最新、收藏、赞过、浏览记录、我的发布和草稿等视图。
- “我的发布”只展示已发布内容，草稿始终单独保存；浏览记录按用户独立持久化，可一键清空。
- 帖子详情支持原生分享链接及复制降级、点赞、收藏、归入收藏夹和添加病例随访。
- 私有收藏夹支持创建、编辑、删除、收录/移出和按专题查看。

#### 互动与社交关系

- 支持评论、回复、评论点赞、编辑、删除、最热/最新排序和问答采纳。
- 支持关注、粉丝/关注名单、公开个人主页、个人作品、发布/粉丝/关注/获赞统计和资料编辑。
- 支持 `@用户名` 提及、互动通知、关注通知、私信提醒和已读状态。
- 私信受关注关系和屏蔽关系保护：普通用户须存在关注关系才能发起，专家/农技人员/管理员可接收首次咨询；被屏蔽后双方不能继续私信或查看对方社区资料。
- 屏蔽管理可查看已屏蔽用户并随时解除屏蔽。

#### 搜索、推荐与治理

- 搜索同时覆盖帖子、用户和话题，可直接进入帖子详情、用户主页或话题流。
- 话题可关注；热门话题按仍公开的真实帖子聚合，过滤无效历史标签。
- 推荐基于浏览、点赞、收藏、评论和关注等轻量行为生成，并展示推荐理由；用户可重置浏览与“不感兴趣”偏好。
- 支持地区筛选、防抖搜索、同作者轻量打散、图片懒加载和接口失败后的乐观更新回滚。
- 支持不感兴趣、举报、专业认证申请及管理员审核；发布、评论、私信具备短时频率限制和重复内容拦截。
- 帖子详情、通知和二次确认使用统一的应用内弹层，位于内容与顶部栏之上，支持 `Esc` 关闭；高风险删除操作使用明确的确认卡片。
- 社区通知与私信以 API 持久化数据为唯一事实来源，通过事件流触发轻量刷新，避免仅依赖前端模拟状态。

### 6. 已形成的数据闭环

1. 用户上传材料或录入养殖现场数据，进入 AI 问诊。
2. 问诊可沉淀为私有病例，或转为待发布社区草稿。
3. 养殖工作台持续记录批次、日记录、病例和随访。
4. 社区发布经验、获得评论和互动后，可将病例内容保存回私人养殖台账。

以上闭环使用真实 API 和 PostgreSQL 业务数据，不以静态演示数据作为唯一来源。

## 用户端体验与视觉规范

- 界面统一采用 Apple 风格：冷白背景、圆角卡片、克制阴影、分层弹窗、蓝色主操作与清晰的危险操作状态。
- 问诊、社区和养殖工作台复用相近的排版、状态、按钮和响应式规则。
- 社区详情、资料卡、收藏夹、随访和删除确认均处于正确的视觉层级，不会被聊天背景、帖子内容、顶部栏或输入框遮挡。
- 窄屏下保持主要阅读、输入和操作路径可用；后续只需按真实设备反馈作微调。

## 明确延期：不计入已完成能力

以下四项由产品范围明确延期。界面占位、设置开关或演示页面均不能视为已接通：

1. **真实 RAG**：知识采集、切分、向量化、召回、重排、引用和版本治理。
2. **真实知识图谱**：实体关系存储、图数据库查询和图谱探索页的数据接入。
3. **可追溯诊断证据**：把检索、规则或图路径绑定到诊断结论，并向用户展示证据来源。
4. **长期记忆**：记忆模型、授权写入、检索、编辑/清除和 Memory Agent。

简历、答辩和演示可将上述内容表述为“后续演进方向”，不得描述为当前成果。

## 正式上线前才需要完成的事项

这些是部署与运行保障，不是当前用户端缺少的业务功能；应在公开上线**之前**完成。

### 外部服务配置

- 配置真实短信或 SMTP 邮件服务，并在生产环境关闭 `CAN_WEN_AUTH_DEV_CODE_ENABLED`。
- 配置 S3 兼容对象存储、私有访问策略与附件生命周期规则。
- 配置真实 OpenAI-compatible 模型地址、密钥、模型配额与故障降级策略。
- 配置 HTTPS、反向代理、生产环境变量和密钥管理。

### 多实例与运维保障

- 使用 Redis 或 API 网关替换进程内限流和事件分发，以支持多实例部署。
- 接入集中日志、错误追踪、指标监控和告警。
- 制定数据库备份、恢复与灾难演练策略。
- 为关键路径固化浏览器端到端测试并接入 CI：登录、问诊发送、问诊转病例、日记录、发帖、评论、收藏、屏蔽、删除确认等流程应在每次变更后自动回归。

## 代码结构与维护建议

- `frontend/src/App.tsx` 和 `frontend/src/styles.css` 仍较大。社区与问诊已开始迁往 `frontend/src/features/community/`、`frontend/src/features/diagnosis/`；后续可继续拆分养殖、项目和设置模块，不改变现有 API。
- 当前已有 `docker-compose.dev.yml`、`RUNBOOK.md` 和 GitHub Actions 的基础构建/测试检查；浏览器端到端用例尚未系统固化到 CI。
- 视频抽帧、转码和异步作业可在真实队列基础设施可用后增强。
- 工作区可能含有用户已有修改；禁止使用 `git reset --hard` 或覆盖式 checkout 清理。

## 管理员端（简要）

管理员端与用户端分离，使用独立 `admin` schema，已具备管理员邀请、角色权限、MFA、审计日志、运营总览、用户处置、社区举报/认证审核、问诊与病例复核、模型与系统设置等能力。用户业务表不得加入管理员迁移的删除计划。
